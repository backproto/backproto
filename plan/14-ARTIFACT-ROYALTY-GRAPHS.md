# Artifact royalty graphs (kind 31930)

## Problem

Agent A builds a prompt template. Agent B uses that template inside a pipeline that serves 10,000 requests. Agent A gets nothing. There is no mechanism to track downstream use of upstream artifacts or to flow payment back through the dependency chain.

## Event schema

Kind 31930 — Artifact royalty declaration:

```
{
  "kind": 31930,
  "content": "<artifact content hash>",
  "tags": [
    ["d", "<artifact_id>"],
    ["artifact_type", "prompt_template|skill_config|pipeline_def|model_weights"],
    ["creator", "<pubkey>"],
    ["royalty_bps", "500"],          // 5% per downstream use
    ["depth_cap", "3"],              // royalties stop propagating after 3 hops
    ["parent", "<parent_artifact_id>"],  // dependency link
    ["license", "MIT|commercial|restricted"]
  ]
}
```

## Royalty terms

- `royalty_bps`: Basis points (1/100th of a percent) of downstream revenue owed to the artifact creator per use.
- `depth_cap`: Maximum number of hops in the dependency DAG through which royalties propagate. Default 3. This prevents infinite chains and keeps settlement tractable.
- Royalties compound multiplicatively: if A charges 5% and B (which depends on A) charges 3%, a consumer of B pays 3% to B and 5% of 97% to A. At depth 3, the numbers stay small.

## Dependency DAG settlement

When a job completes and payment settles:

1. The settlement service traverses the artifact dependency DAG for the skill that executed the job.
2. For each artifact in the DAG (up to `depth_cap` hops):
   - Compute royalty = `payment * (royalty_bps / 10000)` at each hop
   - Subtract already-paid royalties from earlier hops
3. Distribute royalty payments over Lightning using the same settlement rail as the primary payment.

The DAG is constructed from `parent` tags in kind-31930 events. cycles are rejected — if A depends on B and B depends on A, the second declaration fails validation.

## Quality scoring from downstream receipts

An artifact's quality score is derived from completion receipts generated by jobs that used it:

```
artifact_quality = weighted_avg(downstream_completion_scores)
```

Weight = recency. Recent receipts count more. EWMA with alpha=0.3 matches the capacity scoring parameter.

Artifacts with quality below 0.5 get flagged. Routing deprioritizes skills that depend on low-quality artifacts.

## Open questions

- Should royalty rates be negotiable per-consumer, or fixed by the creator?
- Should there be a minimum job payment below which royalty calculation is skipped (to avoid dust payments)?
- How to handle artifact versioning — does a new version reset the quality score?
